# 检索优化设计说明

## 1. 为什么这样设计 Hybrid Search

在RAG（Retrieval-Augmented Generation）系统中，检索质量直接决定了生成答案的准确性。单一的向量检索或关键词检索都存在固有缺陷，混合检索（Hybrid Search）通过融合多种检索方式的优势，显著提升检索的召回率和准确率。

本项目的Hybrid Search设计遵循 **"多路召回 → 融合 → 精排 → 去重"** 的经典架构：

```
用户Query
    │
    ├──→ 向量检索（语义召回，top_k×3）
    │
    ├──→ BM25检索（关键词召回，top_k×3）
    │
    └──→ RRF加权融合
            │
            └──→ Rerank精排
                    │
                    └──→ MMR去重
                            │
                            └──→ 最终结果（top_k）
```

**设计原则**：
- **先宽后窄**：先多路召回更多候选（top_k×3），再逐步精排筛选
- **互补融合**：向量检索和BM25从不同角度理解用户意图，互补性强
- **可控可调**：各环节参数可调（权重、lambda、top_k），适应不同场景

## 2. 向量检索和BM25各自的优势

### 向量检索（Dense Retrieval）

| 优势 | 说明 |
|------|------|
| **语义理解** | 能理解同义词、近义词，如"便宜"和"实惠" |
| **跨语言能力** | 多语言模型可跨语言检索 |
| **上下文感知** | 基于上下文理解词义，如"苹果"在产品语境下指品牌 |
| **模糊匹配** | 对拼写错误、表述差异有一定容忍度 |

| 劣势 | 说明 |
|------|------|
| **精确匹配弱** | 对特定型号、编号等精确匹配不如关键词检索 |
| **黑盒性** | 难以解释为什么某文档被召回 |
| **领域偏移** | 超出训练数据分布时效果下降 |

### BM25检索（Sparse Retrieval）

| 优势 | 说明 |
|------|------|
| **精确匹配** | 对特定关键词、型号、编号匹配精准 |
| **可解释性** | 基于词频统计，结果可解释 |
| **无需训练** | 不依赖预训练模型，零成本部署 |
| **低延迟** | 计算简单，响应速度快 |
| **长尾词优势** | 对低频专业术语匹配效果好 |

| 劣势 | 说明 |
|------|------|
| **无语义理解** | "便宜"和"实惠"无法关联 |
| **词汇鸿沟** | 无法理解同义不同词的表述 |
| **分词依赖** | 中文分词质量直接影响效果 |

### 互补关系

```
场景: 用户问"有没有性价比高的手机？"

向量检索 → 召回: "这款手机价格实惠，配置均衡..." ✓ (理解"性价比"≈"实惠")
BM25检索 → 召回: "性价比高的手机推荐：X200 Pro..." ✓ (精确匹配"性价比")

两者互补，确保不遗漏相关文档
```

## 3. RRF融合算法原理

### Reciprocal Rank Fusion (RRF)

RRF是一种简单而有效的多路检索结果融合方法，核心思想是：**一个文档在多个检索系统中排名越靠前，其综合相关性越高**。

### 公式

```
RRF_score(d) = Σ (w_i / (k + rank_i(d)))
```

其中：
- `d`：候选文档
- `w_i`：第i路检索的权重
- `rank_i(d)`：文档d在第i路检索中的排名（从1开始）
- `k`：平滑参数，默认60

### 参数说明

- **k=60**：标准RRF参数，作用是平滑排名差异。k值越大，排名靠前和靠后的分数差异越小；k值越小，排名靠前的文档优势越明显
- **权重分配**：本项目默认向量检索权重0.6，BM25权重0.4，偏向语义理解但保留关键词匹配能力

### 示例

```
文档A: 向量排名=1, BM25排名=5
文档B: 向量排名=3, BM25排名=1

RRF(A) = 0.6/(60+1) + 0.4/(60+5) = 0.00984 + 0.00615 = 0.01599
RRF(B) = 0.6/(60+3) + 0.4/(60+1) = 0.00952 + 0.00656 = 0.01608

文档B综合排名更高（因为BM25排名第一的加分超过了向量排名的差异）
```

### 为什么选择RRF而非其他融合方法

| 方法 | 优点 | 缺点 |
|------|------|------|
| **RRF** | 无需归一化分数，对分数分布不敏感，实现简单 | 仅考虑排名，忽略分数绝对值 |
| **线性加权** | 利用分数绝对值 | 不同检索系统分数范围不同，需归一化 |
| **CombSUM** | 简单求和 | 分数尺度不一致时效果差 |
| **CombMNZ** | 奖励多路命中 | 对单路命中的好结果不公平 |

RRF在大多数场景下表现稳健，且不需要分数归一化，是工程实践中的首选。

## 4. Rerank的作用

### 为什么需要Rerank

向量检索和BM25检索都是**双塔模型**（Bi-Encoder），query和document分别编码后计算相似度。这种方式速度快，但无法捕捉query和document之间的细粒度交互信息。

Rerank使用**交叉编码器**（Cross-Encoder），将query和document拼接后一起编码，能捕捉更深层的语义交互关系。

### 工作流程

```
粗排阶段（双塔）: query → [编码] → 向量 ← [编码] ← document
                    快速计算，召回大量候选

精排阶段（交叉）: [query ⊕ document] → [CrossEncoder] → 相关性分数
                    精确计算，少量候选上精排
```

### 为什么不直接用CrossEncoder检索

CrossEncoder需要将query和每个document拼接后逐一计算，复杂度为O(N)，无法像向量检索那样利用ANN索引实现O(logN)检索。因此采用**先粗排后精排**的两阶段策略。

### 本项目使用的Reranker

- **模型**：BAAI/bge-reranker-v2-m3
- **特点**：多语言支持，中文效果好，模型体积适中
- **降级策略**：模型加载失败时自动降级为简单关键词匹配评分

## 5. MMR去重的意义

### 问题：检索结果冗余

混合检索后，经常出现多个高度相似的文档被同时召回，例如：

```
结果1: "X200 Pro搭载骁龙8 Gen3处理器，性能强劲"
结果2: "X200 Pro配备了骁龙8 Gen3芯片，运行速度快"
结果3: "X200 Pro使用骁龙8 Gen3，性能表现出色"
```

这三个结果内容高度重复，用户只需要看到一条即可。

### MMR算法

MMR（Maximal Marginal Relevance）在选择每个结果时，同时考虑**与query的相关性**和**与已选结果的差异性**：

```
MMR(d) = λ × Sim(d, q) - (1-λ) × max[Sim(d, d_j)] for d_j in 已选集合
```

- `λ=1.0`：只看相关性，等同于普通排序
- `λ=0.0`：只看多样性，选择最不同的文档
- `λ=0.7`（默认）：偏向相关性但兼顾多样性

### 效果

```
MMR前: [相似文档1, 相似文档2, 相似文档3, 不同文档4, 不同文档5]
MMR后: [相似文档1, 不同文档4, 不同文档5, ...]
```

确保返回的结果既相关又多样化，提升用户体验。

## 6. 性能提升预估（相比纯向量检索）

| 指标 | 纯向量检索 | Hybrid Search | 提升 |
|------|-----------|---------------|------|
| **检索延迟** | ~50ms | ~150ms（含BM25+Rerank） | 延迟增加约100ms |
| **吞吐量** | 高 | 中等 | 约降低30% |
| **内存占用** | FAISS索引 | FAISS+BM25+Reranker | 增加约500MB（Reranker模型） |

**说明**：混合检索在性能上有一定开销，但对于售前客服场景（QPS不高、对准确率要求高），这点性能损失完全可以接受。可以通过以下方式优化：
- BM25索引预加载到内存
- Reranker使用ONNX Runtime加速推理
- 异步并行执行向量检索和BM25检索

## 7. 准确率提升预估

基于业界实践和学术论文的参考数据：

| 指标 | 纯向量检索 | Hybrid Search | 提升 |
|------|-----------|---------------|------|
| **Recall@5** | 72-78% | 85-92% | +10-20% |
| **Precision@5** | 65-72% | 78-88% | +10-18% |
| **MRR** | 0.68-0.75 | 0.82-0.90 | +10-18% |
| **NDCG@5** | 0.70-0.76 | 0.83-0.91 | +12-20% |

**关键提升场景**：

1. **精确匹配场景**（产品型号、编号）：纯向量检索容易遗漏，BM25补充后Recall提升20%+
2. **同义表述场景**（"便宜"vs"实惠"）：BM25无法关联，向量检索补充后Recall提升15%+
3. **Rerank精排**：粗排结果中相关性排序不准确的问题，精排后Precision提升10-15%
4. **MMR去重**：减少冗余结果，有效信息密度提升20-30%
